# 非功能需求 — AgentForge

## 性能

**NFR-1**: 在基础镜像已缓存且国内镜像源可用的网络条件下，系统必须在 15 分钟内完成 `build -d all --max-retry 3` 命令的完整构建过程，退出码为 0。

> 来源：PRD — 成功标准（各 build 退出码为 0，镜像在 docker images 中可见）、PRD 风险（网络不稳定, 镜像体积过大）

**NFR-2**: 使用 `-d mini` 构建的镜像体积必须小于 `-d all` 构建的镜像体积的 60%。

> 来源：PRD — 成功标准（"mini 镜像体积小于 all"）、PRD 风险（"镜像体积过大，占用大量磁盘空间"）

**NFR-3**: 系统必须在 10 秒内完成从执行 `run -a <agent>` 命令到容器内交互式终端提示符可接受输入的过程。

> 来源：PRD — 成功标准（run 命令启动 agent 交互终端）、Story 3 验收标准

**NFR-4**: 当 `endpoint test <name>` 连接的 LLM 端点不可达（连接超时、DNS 解析失败或连接被拒绝）时，系统必须在 30 秒内超时断开，返回非零退出码和描述失败原因的错误信息。

> 来源：REQ-27（端点测试失败时输出错误信息并返回非零退出码）、BDD 场景"测试端点连通性失败"、Story 11 验收标准

**NFR-5**: 在执行无需网络或 Docker 操作的非构建类命令（`version`、`--help`、`endpoint list`、`endpoint providers`）时，系统必须在 1 秒内完成全部输出。

> 来源：PRD 目标（各命令的 help 和 version 功能）、整体 CLI 用户体验

## 安全性

**NFR-6**: 系统必须在 `endpoint show <name>` 以及其他任何可能输出 API key 的场景中将密钥内容脱敏，脱敏格式为前 8 位字符加 `***` 加后 4 位字符；不得在任何输出场景中显示完整的 API key。

> 来源：PRD — 风险（"API key 在配置文件中明文存储，可能泄露"）、REQ-21（key 以掩码格式显示）、Story 10 验收标准

**NFR-7**: 系统必须仅当开发者显式传入 `--docker` 或 `--dind` 参数时才能以特权模式和 root 用户启动容器；未传入这些参数时，`run` 命令必须以非特权模式启动容器。

> 来源：PRD — 风险（"容器特权模式可能带来安全风险，--docker/--dind 为显式参数，不会自动启用"）、REQ-15

**NFR-8**: 通过 `-m` 参数挂载宿主机目录时，系统必须在容器内默认以只读权限挂载，容器内进程不得修改宿主机文件系统中的内容。

> 来源：PRD — 风险（"容器内挂载的宿主机目录被意外修改或删除，默认 `run -m` 挂载为只读"）、REQ-11

**NFR-9**: 系统在创建或修改端点配置文件时，必须将文件权限设置为仅文件所有者可读写（`0600`），防止其他系统用户读取 API key 等敏感配置信息。

> 来源：PRD — 风险（"配置文件位于用户私有目录，非共享目录"、"API key 在宿主机和容器配置文件中明文存储"）

## 可靠性

**NFR-10**: 构建过程中发生网络错误时，系统必须按指数退避策略自动重试：首次重试等待至少 1 秒，后续每次等待时间翻倍（1s, 2s, 4s...），总重试次数不超过 `--max-retry` 参数指定的值（默认 3 次）。

> 来源：PRD — 风险（"网络不稳定或镜像源不可用，内部按指数退避策略重试"）、REQ-4（指数退避重试，默认 3 次）、BDD 场景"构建过程中网络错误时自动重试"

**NFR-11**: 当 `-R/--rebuild` 构建过程失败时，系统必须清理构建过程中产生的临时镜像标签，并确保原镜像标签和内容保持不变，退出码非零。

> 来源：REQ-8（重建失败清理临时标签，保留原镜像）、Story 2 验收标准、BDD 场景"重建失败时保留旧镜像"

**NFR-12**: 每次 `run` 命令执行成功后，系统必须将全部运行参数（agent、端口映射、目录挂载、环境变量、工作目录等）完整持久化到 `.last_args` 文件，确保 `-r/--recall` 能恢复与上次运行完全一致的参数集。

> 来源：REQ-16（从 .last_args 恢复参数）、Story 6 验收标准（"每次 run 命令执行后自动将全部参数持久化"）、BDD 场景"通过 -r 参数恢复上次运行参数"

**NFR-13**: 执行 `update` 命令时，系统必须在下载和安装新版本之前备份当前版本；如果更新过程中发生任何错误（下载失败、写入失败、完整性校验失败），系统必须自动回滚到备份版本，确保工具始终保持可运行状态。

> 来源：PRD — 风险（"自更新失败导致 CLI 工具损坏或版本回退无门，更新前保留旧版本的备份"）

## 可用性

**NFR-14**: 当 `endpoint add <name>` 缺少必要参数时，系统必须逐一提示缺失的配置项，每个提示必须包含参数名称、预期的值格式和可选值列表（如 `--provider` 的可选值：deepseek / openai / anthropic）。

> 来源：REQ-23（缺参时交互式逐一提示）、Story 8 验收标准（"交互式方式逐个提问收集缺失的配置项"）、BDD 场景"缺少参数时交互式新增 LLM 端点"

**NFR-15**: 系统每个命令和子命令的 `--help` 输出必须使用统一的格式，至少包含：命令名称和简短用途、参数列表（每个参数含名称、类型、默认值）、至少一个使用示例；格式在所有命令中保持一致。

> 来源：REQ-37（"格式一致的帮助信息"）、Story 16 验收标准（"每个命令和子命令输出格式一致的帮助文本"）、BDD 场景"工具自更新和版本信息查看"

**NFR-16**: 命令执行失败时，系统输出的错误信息必须包含以下三个要素：失败的具体原因（如"连接超时"、"文件未找到"、"参数无效"）、涉及的命令和参数上下文、建议的修正措施或排查方向。

> 来源：REQ-27（端点测试失败输出明确错误信息）、REQ-17（.last_args 不存在时提示无法恢复）、Story 11 验收标准（"输出明确的错误信息"）

## 兼容性

**NFR-17**: 系统必须在 Docker Engine 版本 >= 20.10 的环境下正常运行所有容器化操作（build、run、export、import、deps），不得因 Docker 版本导致的 API 不兼容而失败。

> 来源：PRD — 依赖（"Docker Engine 版本 >= 20.10"）、PRD — 风险（"Docker Engine 版本过旧导致构建行为异常"）

**NFR-18**: 构建过程中系统必须自动检测 Docker 环境是否支持 BuildKit；当 `DOCKER_BUILDKIT=1` 时自动启用 BuildKit 构建加速，否则回退到标准构建模式并在诊断输出中提示用户 BuildKit 状态。

> 来源：PRD — 依赖（"BuildKit 支持，检测 DOCKER_BUILDKIT 环境变量以启用 BuildKit 构建加速"）、PRD — 风险（"缺少 BuildKit 支持导致构建行为异常"）

**NFR-19**: 在 `doctor` 命令自动安装缺失核心依赖时，系统必须正确识别当前操作系统的包管理器（优先顺序：apt-get、dnf、yum、brew），并使用对应当前发行版的命令和参数安装依赖，确保安装后依赖可被系统正常调用。

> 来源：REQ-32（自动使用 apt-get、dnf、yum 或 brew 安装）、Story 13 验收标准（"使用 apt-get / dnf / yum / brew 自动安装缺失的依赖"）、BDD 场景"环境诊断"

**NFR-20**: 构建过程中系统必须自动将 npm 源切换为 npmmirror、pip 源切换为阿里云 PyPI 镜像、yum 源切换为阿里云 CentOS Vault 镜像；当镜像源不可达时，系统必须自动回退到官方源并在构建日志中输出可感知的提示信息。

> 来源：REQ-3（自动使用国内镜像源加速）、PRD — 依赖（"国内镜像源可用性"）、Story 1 验收标准（"自动使用国内镜像源加速"）

## 可维护性

**NFR-21**: `update` 命令成功执行后，`version` 命令输出的版本号和 git hash 必须与实际更新的新版本代码一致；不得出现 `version` 显示的版本号与已更新的二进制产物间的不一致情况。

> 来源：REQ-36（从 Git remote 或 UPDATE_URL 下载更新，嵌入新的 git hash 并更新版本号）、Story 16 验收标准

**NFR-22**: 每次构建 CLI 二进制产物时，系统必须将构建时当前仓库 HEAD commit 的完整 git hash 嵌入到产物中；`version` 命令输出的 git hash 必须能唯一对应源码仓库中的确切的 commit，支持使用该 hash 追溯构建时的源码状态。

> 来源：REQ-36（version 命令输出格式化的版本号和当前 git hash）、Story 16 验收标准

**NFR-23**: 执行 `-R/--rebuild` 重建镜像时，系统必须使用临时标签进行构建，构建成功后以原子操作将临时标签替换为原镜像标签并清理旧镜像；构建失败时必须自动清理临时标签，确保 `docker images` 列表中不残留任何由本次失败构建产生的中间镜像。

> 来源：REQ-7（临时标签构建成功后替换原标签）、REQ-8（重建失败则清理临时标签）、Story 2 验收标准（"使用临时标签构建，构建成功后替换原镜像标签并删除旧镜像"）
