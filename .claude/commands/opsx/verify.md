验证实现是否与变更产物（specs、tasks、design）匹配。

**输入**：可选地在 `/opsx:verify` 之后指定变更名称（如：`/opsx:verify add-auth`）。如果省略，检查是否可以从会话上下文中推断。如果模糊或歧义，你必须提示用户可用的变更。

**步骤**

1. **如果未提供变更名称，提示用户选择**

   运行 `openspec list --json` 获取可用变更。使用 **AskUserQuestion 工具**让用户选择。

   显示有实现任务的变更（tasks 产物存在）。
   如果可用，包含每个变更使用的 schema。
   将有未完成任务的变更标记为"（进行中）"。

   **重要**：不要猜测或自动选择变更。始终让用户选择。

2. **检查状态以了解 schema**
   ```bash
   openspec status --change "<name>" --json
   ```
   解析 JSON 以了解：
   - `schemaName`：正在使用的工作流（如："spec-driven"）
   - 此变更存在哪些产物

3. **获取变更目录并加载产物**

   ```bash
   openspec instructions apply --change "<name>" --json
   ```

   返回变更目录和上下文文件。从 `contextFiles` 读取所有可用产物。

4. **初始化验证报告结构**

   创建一个具有三个维度的报告结构：
   - **完整性**：跟踪任务和 spec 覆盖率
   - **正确性**：跟踪需求实现和场景覆盖率
   - **一致性**：跟踪设计遵循度和模式一致性

   每个维度可以有 CRITICAL、WARNING 或 SUGGESTION 级别的问题。

5. **验证完整性**

   **任务完成**：
   - 如果 tasks.md 存在于 contextFiles 中，读取它
   - 解析复选框：`- [ ]`（未完成）vs `- [x]`（已完成）
   - 计算已完成与总任务数量
   - 如果存在未完成的任务：
     - 为每个未完成的任务添加 CRITICAL 问题
     - 建议："完成任务：<描述>" 或 "如果已实现则标记为已完成"

   **Spec 覆盖率**：
   - 如果 delta specs 存在于 `openspec/changes/<name>/specs/`：
     - 提取所有需求（标记为 "### Requirement:"）
     - 对每个需求：
       - 在代码库中搜索与需求相关的关键词
       - 评估实现是否可能存在
     - 如果需求似乎未实现：
       - 添加 CRITICAL 问题："未找到需求：<需求名称>"
       - 建议："实现需求 X：<描述>"

6. **验证正确性**

   **需求实现映射**：
   - 对 delta specs 中的每个需求：
     - 在代码库中搜索实现证据
     - 如果找到，记录文件路径和行范围
     - 评估实现是否匹配需求意图
     - 如果检测到差异：
       - 添加 WARNING："实现可能与 spec 有差异：<详情>"
       - 建议："审查 <文件>:<行> 对照需求 X"

   **场景覆盖率**：
   - 对 delta specs 中的每个场景（标记为 "#### Scenario:"）：
     - 检查代码中是否处理了这些条件
     - 检查是否存在覆盖该场景的测试
     - 如果场景似乎未被覆盖：
       - 添加 WARNING："场景未覆盖：<场景名称>"
       - 建议："为场景添加测试或实现：<描述>"

7. **验证一致性**

   **设计遵循度**：
   - 如果 design.md 存在于 contextFiles 中：
     - 提取关键决策（查找 "Decision:"、"Approach:"、"Architecture:" 等 section）
     - 验证实现是否遵循这些决策
     - 如果检测到矛盾：
       - 添加 WARNING："设计决策未被遵循：<决策>"
       - 建议："更新实现或修改 design.md 以匹配现实"
   - 如果没有 design.md：跳过设计遵循度检查，注明 "没有 design.md 可供验证"

   **代码模式一致性**：
   - 审查新代码是否与项目模式一致
   - 检查文件命名、目录结构、编码风格
   - 如果发现显著偏差：
     - 添加 SUGGESTION："代码模式偏差：<详情>"
     - 建议："考虑遵循项目模式：<示例>"

8. **生成验证报告**

   **摘要记分卡**：
   ```
   ## 验证报告：<change-name>

   ### 摘要
   | 维度     | 状态              |
   |--------------|------------------|
   | 完整性 | X/Y 任务，N 个需求|
   | 正确性  | M/N 个需求已覆盖 |
   | 一致性    | 已遵循/有问题  |
   ```

   **按优先级排列的问题**：

   1. **CRITICAL**（归档前必须修复）：
      - 未完成的任务
      - 缺失的需求实现
      - 每个都有具体、可操作的建议

   2. **WARNING**（应该修复）：
      - Spec/设计差异
      - 缺失的场景覆盖率
      - 每个都有具体建议

   3. **SUGGESTION**（最好修复）：
      - 模式不一致
      - 小的改进
      - 每个都有具体建议

   **最终评估**：
   - 如果有 CRITICAL 问题："发现 X 个严重问题。在归档前修复。"
   - 如果只有警告："没有严重问题。Y 个警告需要考虑。可以归档（含注明的改进）。"
   - 如果全部通过："所有检查通过。可以归档。"

**验证启发式**

- **完整性**：关注客观的检查清单项（复选框、需求列表）
- **正确性**：使用关键词搜索、文件路径分析、合理推断 — 不需要绝对的确定性
- **一致性**：寻找明显的矛盾，不要对风格吹毛求疵
- **误报**：不确定时，优先使用 SUGGESTION 而非 WARNING，WARNING 而非 CRITICAL
- **可操作性**：每个问题必须有具体的建议，尽可能包含文件/行引用

**优雅降级**

- 如果只有 tasks.md：仅验证任务完成，跳过 spec/design 检查
- 如果有 tasks + specs：验证完整性和正确性，跳过 design
- 如果有完整产物：验证所有三个维度
- 始终注明跳过了哪些检查以及为什么

**输出格式**

使用清晰的 markdown：
- 摘要记分卡使用表格
- 问题按组列表（CRITICAL/WARNING/SUGGESTION）
- 代码引用格式：`file.ts:123`
- 具体、可操作的建议
- 没有像"考虑审查"这样的模糊建议
