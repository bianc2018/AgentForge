实现 OpenSpec 变更中的任务。

**输入**：可选地在 `/opsx:apply` 之后指定变更名称（如：`/opsx:apply add-auth`）。如果省略，检查是否可以从会话上下文中推断。如果模糊或歧义，你必须提示用户可用的变更。

**步骤**

1. **选择变更**

   如果提供了名称，使用它。否则：
   - 如果用户在会话中提到过变更，从对话上下文中推断
   - 如果只有一个活跃的变更，自动选择
   - 如果歧义，运行 `openspec list --json` 获取可用变更，并使用 **AskUserQuestion 工具**让用户选择

   始终宣布："正在使用的变更：<name>" 以及如何覆盖（如：`/opsx:apply <other>`）。

2. **检查状态以了解 schema**
   ```bash
   openspec status --change "<name>" --json
   ```
   解析 JSON 以了解：
   - `schemaName`：正在使用的工作流（如："spec-driven"）
   - 哪个产物包含任务（spec-driven 通常是 "tasks"，其他情况请检查状态）

3. **获取 apply 说明**

   ```bash
   openspec instructions apply --change "<name>" --json
   ```

   返回：
   - 上下文文件路径（根据 schema 不同而不同）
   - 进度（总数、已完成、剩余）
   - 带状态的任务列表
   - 基于当前状态的动态指令

   **处理状态：**
   - 如果 `state: "blocked"`（产物缺失）：显示消息，建议使用 `/opsx:continue`
   - 如果 `state: "all_done"`：祝贺，建议归档
   - 否则：继续实现

4. **读取上下文文件**

   从 apply 说明输出中读取 `contextFiles` 中列出的文件。
   文件取决于正在使用的 schema：
   - **spec-driven**：proposal、specs、design、tasks
   - 其他 schema：遵循 CLI 输出中的 contextFiles

5. **显示当前进度**

   显示：
   - 正在使用的 Schema
   - 进度："N/M 个任务已完成"
   - 剩余任务概览
   - CLI 的动态指令

6. **实现任务（循环直到完成或被阻塞）**

   对于每个待处理的任务：
   - 显示正在处理哪个任务
   - 进行所需的代码更改
   - 保持更改最小化和聚焦
   - 在任务文件中标记任务完成：`- [ ]` → `- [x]`
   - 继续下一个任务

   **暂停情况：**
   - 任务不明确 → 要求澄清
   - 实现揭示了设计问题 → 建议更新产物
   - 遇到错误或阻塞 → 报告并等待指导
   - 用户中断

7. **完成或暂停时，显示状态**

   显示：
   - 本次会话完成的任务
   - 总体进度："N/M 个任务已完成"
   - 如果全部完成：建议归档
   - 如果暂停：解释原因并等待指导

**实现过程中的输出**

```
## 正在实现：<change-name>（schema：<schema-name>）

正在处理任务 3/7：<任务描述>
[...正在进行实现...]
✓ 任务完成

正在处理任务 4/7：<任务描述>
[...正在进行实现...]
✓ 任务完成
```

**完成时输出**

```
## 实现完成

**变更：** <change-name>
**Schema：** <schema-name>
**进度：** 7/7 个任务完成 ✓

### 本次会话完成
- [x] 任务 1
- [x] 任务 2
...

所有任务已完成！你可以使用 `/opsx:archive` 归档此变更。
```

**暂停时输出（遇到问题）**

```
## 实现已暂停

**变更：** <change-name>
**Schema：** <schema-name>
**进度：** 4/7 个任务完成

### 遇到的问题
<问题的描述>

**选项：**
1. <选项 1>
2. <选项 2>
3. 其他方法

你想怎么做？
```

**约束**
- 持续处理任务直到完成或被阻止
- 开始前始终读取上下文文件（来自 apply 说明输出）
- 如果任务不明确，在实现前暂停并询问
- 如果实现揭示问题，暂停并建议产物更新
- 保持代码更改最小化并限定在每个任务范围内
- 每个任务完成后立即更新任务复选框
- 遇到错误、阻塞或不明确的需求时暂停 — 不要猜测
- 使用 CLI 输出中的 contextFiles，不要假设特定文件名

**灵活的工作流集成**

此 skill 支持"对变更的操作"模型：

- **可以随时调用**：在所有产物完成之前（如果存在任务）、部分实现之后、与其他操作交错
- **允许产物更新**：如果实现揭示了设计问题，建议更新产物 — 不是阶段锁定的，灵活工作
