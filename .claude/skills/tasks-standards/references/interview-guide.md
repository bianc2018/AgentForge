# 索引细化指南 — Tasks

## 何时使用本指南

本指南仅在**步骤 1**中使用 — 当 agent 提出索引并等待
确认时。步骤 2（生成）没有问题：所有内容从产物中推导。

---

## 提出索引前要检查的内容

在向用户呈现索引之前，内部验证：

**粒度：**
- 有任何 REQ 块只有一个"实现组件 X"的 task 而实际上涵盖 3 个独立方法？ → 拆分
- 有任何 task 太小而不能独立产生价值且总是与另一个一起？ → 合并

**覆盖率：**
- 每个 REQ 至少有 1 个 task？每个 NFR？每个 Scenario？
- 外部服务 adapters（邮件、短信）有独立的 task？
- 每个有载荷的 endpoint 都有验证 schema task？

**分组：**
- 每个 NFR 是否在其 `来源` 字段指向的 REQ 块中？
- 测试 task 是否在 Scenario 覆盖的 REQ 块中？

---

## 用于向用户提问的问题

仅在识别到真正的空白时使用 — 不要问产物中已有的内容。

### 关于范围
- "设计提到了 `<外部服务>` — 我是否应该为其包含一个凭证/环境变量配置的 task？"
- "`doc/constitution.md` 要求 `<规则>`。是否有任何验证产物（lint、架构测试）应该成为 task？"

### 关于粒度
- "`<组件X>` 有方法 `save()`、`findValid()` 和 `markAsUsed()`。你希望每个方法一个 task 还是整个组件一个 task？"
- "Task T-XX 和 T-YY 实现了始终由同一开发人员一起完成的行为。合并它们有意义吗？"

### 关于分组
- "NFR-N 没有直接引用任何 REQ。我应该将其放在独立的 NFR 块中还是将其关联到看起来最接近的 REQ-N？"

---

## 如何纳入用户调整

| 请求 | 如何操作 |
|--------|-----------|
| "添加 X 的 task" | 插入到正确的块中，分配临时 ID，更新总数 |
| "删除 task Y" | 删除并检查是否有其他 task 依赖它 |
| "合并 X 和 Y" | 合并为单个项目，保留较小的 ID，丢弃较大的 |
| "将 X 拆分为两个" | 添加第二个 ID，调整依赖关系 |
| "将 X 移动到 REQ-N 的块" | 移动并检查分组是否仍然合理 |

纳入后，显示完整的修订索引并在推进前确认。

---

## 什么不要问

- 可追溯性 — 从产物中推导，在生成步骤中填写
- 估算 — 不在 tasks.md 的范围内
- 负责人 — 不在 tasks.md 的范围内
- 设计或需求中已经明确的内容
