# 访谈指南 — PRD 创建

## 访谈原则

1. **一次一个问题。** 绝不在同一条消息中问两个问题。用户需要回答多件事时会失去思路。

2. **开放式问题，非确认性问题。** 优先使用"谁会使用这个？"而非"配送员会使用这个吗？"。是/否回答很少能揭示缺失的内容。

3. **使用累积的上下文。** 每个 section 应引用已经讨论过的内容。如果用户在概述中提到了"已注册的配送员"，不要从零开始问目标用户 — 进行上下文关联："你提到了已注册的配送员。除了注册之外，他们还需要满足什么前置条件吗？"

4. **不要问显而易见的事。** 如果 feature 是"配送员登录"，不要问"谁来做登录？"。生成一个合理的草稿，只问真正缺失的内容。

5. **模糊时深入挖掘。** 如果回答是"正常工作"或"常规做法"，要求一个具体的例子或具体的数字。

---

## 何时接受 vs. 深入挖掘

**接受**当：
- Section 检查清单中的所有项目已覆盖
- 信息具体且可验证
- 生成的草稿没有明显空白

**深入挖掘**当：
- 回答模糊（"能用的东西"、"常规做法"、"看情况"）
- 检查清单中至少缺失一个必需项目
- 回答与之前的 section 矛盾或冲突
- 用户描述解决方案而非问题（特别是在问题和目标 section 中）

## 各 section 的问题库

### 1. 概述

当草稿不清楚或不完整时使用。

- "一句话：配送员通过这个 feature 能做到什么今天做不到的事？"
- "系统用来实现这个的主要机制是什么？（如：token、会话、短信验证码）"
- "有没有什么定义其工作原理的关键技术词在草稿中不够清楚？"

### 2. 问题

当问题被描述为解决方案，或影响不清晰时使用。

- "当配送员今天尝试做 [feature 的操作] 时会发生什么？"
- "这对配送员的日常工作的具体后果是什么？"
- "为什么这个问题需要现在解决，而不能放在其他 feature 中？"

### 3. 目标用户

当目标用户过于通用或存在隐含前置条件时使用。

- "任何配送员都可以使用这个，还是有什么先决条件？（如：需要已创建账户、需要通过某个步骤）"
- "有没有*不应该*拥有此 feature 访问权限的配送员？"

### 4. 目标

当目标模糊、包含弱动词（"是"、"有"、"保证"）或不足时使用。

- "除了 [已列出的目标]，系统还需要做什么来交付这个 feature？"
- "当 [特定场景] 时，系统具体做什么？这是一个独立的目标吗？"
- "目标 '[X]' 可验证吗？我们如何知道它被正确实现了？"

### 5. 成功标准

当标准不可测量或不覆盖错误场景时使用。

- "我们如何知道正常路径成功了？配送员看到或收到什么？"
- "当凭据错误时会发生什么？这需要成为一个成功标准吗？"
- "有没有性能或安全要求需要在这个列表中？"

### 6. 不在范围内

当范围模糊或用户没有考虑什么要排除时使用。

- "配送员可能合理地从该 feature 期待什么，但这次不会交付？"
- "有没有类似的功能（如：登出、密码恢复）可能与此混淆？它是否明确排除在外？"

### 7. 主流程

当缺少步骤、入口点不清楚或流程有分支时使用。

- "流程从哪里开始？配送员打开特定的 URL，还是从其他地方过来？"
- "在 [步骤 X] 之后具体发生什么？系统接下来做什么？"
- "流程在哪里结束？配送员最终看到或收到什么？"

### 8. 替代流程

当存在相关的辅助流程但未被提及时使用。

- "有没有在后台或自动发生的流程？（如：token 续期、重试）"
- "如果配送员已经认证并再次尝试访问该 feature，会发生什么？"
- "这个 feature 没有替代流程吗？我们可以标记为 'N/A' 并附上简短理由。"

### 9. 依赖

当隐含的依赖未被提及时使用。

- "这个 feature 是否依赖于其他必须先存在的产品功能？"
- "有没有特定的外部库或服务将被使用？"
- "有没有必须先存在的数据库结构（表、列）？"

### 10. 风险

当没有列出风险或风险过于通用时使用。

- "这个 feature 可能出现什么问题会损害配送员或平台安全？"
- "我们如何缓解 [你在 feature 上下文中识别的特定风险]？"
- "这个 feature 有没有安全风险？（认证、敏感数据、不可逆操作）"

## 警告信号 — 需要深入挖掘的回答

| 收到的回答 | 问题 | 如何深入挖掘 |
|---|---|---|
| "正常工作" | 不可测量 | "在这种情况下'正常'是什么意思？什么是成功的例子？" |
| "用户们"（未指定） | 目标用户通用 | "具体是哪种类型的用户？配送员、管理员、客户？" |
| "所有必要的"（不在范围内） | 未界定任何内容 | "如果让你列出 3 个某人可能期待但不会交付的东西，会是什么？" |
| "基本就是正常流程" | 流程未描述 | "你能逐步描述这个流程吗，从用户首先做什么开始？" |
| "取决于实现"（在目标中） | 目标过于技术化 | "从用户可见行为的角度来说，有什么变化？" |
| 在"问题"中描述解决方案 | 问题/解决方案颠倒 | "在考虑解决方案之前：今天当用户尝试这样做时会发生什么？" |

---

## 访谈节奏

- 始终宣布你在哪个 section：`[Section 3/10：目标用户]`
- 在提问之前展示生成的草稿
- 纳入回答后，展示更新的草稿并在进一步提问或推进之前大声评估
- 当 section 准备好时，确认："Section 已完成。前进到 [下一个 section]。"
- 如果用户想跳过某个 section，接受但记录为"N/A — [用户给出的理由]"
