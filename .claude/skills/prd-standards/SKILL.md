# PRD 标准

## 格式规范

> 如果项目中存在 `docs/sdd/prd-format.md`，读取它并将其用作格式规范 — 它替代以下默认格式，可能已为此项目定制。

## 默认格式

### 必需结构（10 个 section，按此顺序）

1. 概述
2. 问题
3. 目标用户
4. 目标
5. 成功标准
6. 不在范围内
7. 主流程
8. 替代流程
9. 依赖
10. 风险

---

## 各 section 检查清单

### 1. 概述
_格式：连续文本，1-3 句话。_
- [ ] 回答"什么"（用户可以做什么）
- [ ] 回答"如何"（主要机制，如：JWT、httpOnly cookie）
- [ ] 无实现细节（函数名、表结构）
- [ ] 不以元语句开头（"本 PRD 描述了..."）

### 2. 问题
_格式：连续文本，1-3 段。_
- [ ] 阐述当前状况（没有此 feature 时会发生什么）
- [ ] 解释对用户的具体影响
- [ ] 不描述解决方案

### 3. 目标用户
_格式：1-2 句具体描述。_
- [ ] 具体（不是"任何用户"）
- [ ] 如有前置条件则提及（如："通过 register-user 预先创建了账户"）

### 4. 目标
_格式：编号列表，每项以动词不定式开头。_
- [ ] 3 到 7 项
- [ ] 每项可验证（可以判断"已实现或未实现"）
- [ ] 没有项目重复或重叠
- [ ] 不混淆目标与成功标准

### 5. 成功标准
_格式：`标准 | 度量` 表格。_
- [ ] 3 到 6 行
- [ ] "度量"列可量化或可验证（%、时间、布尔值）
- [ ] 至少覆盖：正常路径、主要错误和一项非功能需求

### 6. 不在范围内
_格式：bullet 列表。_
- [ ] 至少 3 项
- [ ] 用户可能合理期望但不会交付的项目
- [ ] 没有明显永远不会在范围内的项目

### 7. 主流程
_格式：`flowchart LR` Mermaid 图，每个节点是一个不同的操作或决策。_
- [ ] 从用户入口点开始
- [ ] 以最终结果结束
- [ ] 没有分支（错误放在替代流程中）
- [ ] 每个步骤是一个不同的操作或决策
- [ ] 使用 `flowchart LR`，节点使用产品语言描述（无代码）

### 8. 替代流程
_格式：与主流程相同的格式（`flowchart LR`）。每个不同的替代流程一个子图。可以是带理由的"N/A"。_
- [ ] 如果产品需要保证有辅助流程，则必须存在
- [ ] 与主流程明显不同
- [ ] 相同的详细程度

### 9. 依赖
_格式：bullet 列表。_
- [ ] 提及需要预先存在的产品功能
- [ ] 提及非平凡的外部库
- [ ] 提及需要预先存在的数据或数据库结构
- [ ] 没有明显通用的依赖（如："Next.js"、"数据库"）

### 10. 风险
_格式：`风险 | 缓解` 表格。_
- [ ] 2 到 5 行
- [ ] 每个风险特定于此 feature
- [ ] 每个缓解措施具体且可操作
- [ ] 如果 feature 涉及认证或敏感数据，至少覆盖一个安全风险

---

## 格式规则

- **语气：** 面向产品。PM 不需要懂代码就能理解。
- **语言：** 中文。成熟的技术术语（JWT、OAuth、bcrypt）保持英文。
- **标题：** `# PRD — <Feature 名称>`（使用破折号，非连字符）
- **空 section：** 绝不留空 — 如果不适用，写明原因。
