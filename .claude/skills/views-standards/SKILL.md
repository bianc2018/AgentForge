# 屏幕（视图）文档标准

## 格式规范

> 如果项目中存在 `docs/sdd/views-format.md`，读取它并将其用作格式规范 — 它替代以下默认格式，可能已为此项目定制。

## 默认格式

### `tela.md` 的必需结构

每个文件严格遵循这 **5 个 section**，按此顺序：

```markdown
# <屏幕显示名称>

## 概述

## 组件

## 状态

## 考量

## 视觉参考
```

---

## Section 1 — 概述

**目的：** 明确标识屏幕并将其目的置于 feature 流程中。

**必需字段：**

| 字段 | 描述 |
|---|---|
| **名称** | 屏幕显示名称，使用中文，如场景或 PRD 中所示 |
| **Slug** | kebab-case 规范标识符，无重音（如：`pagina-de-cadastro`） |
| **URL** | 路由路径（如：`/register`）。如果产物未确定则使用 `—` |
| **目的** | 1–2 句面向用户的描述：用户在此屏幕做什么/能做什么 |
| **相关 Scenarios** | 涉及此屏幕的 `.feature` 中的**确切**标题列表 |

**质量检查清单：**
- [ ] 名称与 Scenarios 和 PRD 中出现的名称一致
- [ ] 目的面向用户 — 不描述实现
- [ ] 所有提及此屏幕的 Scenarios 已列出
- [ ] URL 从产物中的显式片段推导（或标记为 `—`）

---

## Section 2 — 组件

**目的：** 描述屏幕的所有视觉和交互元素。

**必需的子 section（仅包含适用的）：**

### 表单字段

从 Scenarios 中识别的输入字段表：

| 字段 | 类型 | 必填 | 描述 |
|---|---|---|---|
| 字段名称 | text / email / password / date / file / 等 | 是 / 否 | 用户输入或选择的内容 |

### 按钮

按钮和操作控件表：

| 标签 | 类型 | 行为 |
|---|---|---|
| UI 中显示的按钮文本 | submit / button / link-button | 点击时发生什么 |

### 导航链接

指向其他屏幕的链接表：

| 标签 | 目标 | 上下文 |
|---|---|---|
| 链接文本 | URL 或目标屏幕名称 | 何时/何地出现 |

### 固定内容和消息

与用户相关的静态文本（标题、副标题、说明）：

- `<确切文本或内容描述>`

**质量检查清单：**
- [ ] Scenarios 中提及的所有字段都在表中（如：`Then 系统显示一个包含姓名、邮箱...字段的表单`）
- [ ] 按钮标签如 UI 中所示，非技术名称
- [ ] 按钮行为从 Scenarios 的 `When` 和 `Then` 推导
- [ ] 导航链接从 Scenarios 的 `Then ... 被带到` 推导
- [ ] 无库组件名称（不使用 `Input`、`Button`、`Link` — 使用"文本字段"、"提交按钮"、"链接"）
- [ ] 无源代码或组件 props

---

## Section 3 — 状态

**目的：** 描述屏幕在每个可观察情况下的行为。

**格式：** 在描述每个状态之前，包含一个 `stateDiagram-v2` 图，表示屏幕状态机的所有转换。

**要记录的状态（仅包含适用的）：**

### 默认（initial）
用户首次访问时看到的内容。预填字段、初始焦点、欢迎消息。

### 加载（loading）
用户等待系统响应时看到的内容。适用于有异步操作时（表单提交、查询、上传）。

### 错误
从 Scenarios 的 `Then` 中复制的**原文错误消息**，按原因组织：

| 原因 | 显示的消息 |
|---|---|
| 错误条件描述 | `"如 .feature 中写的确切消息"` |

### 成功
用户在成功操作后看到或收到的内容。从正常路径的 `Then` 推导。

### 空（如适用）
没有数据可显示时的状态（列表、历史记录、搜索结果）。

**质量检查清单：**
- [ ] 包含 `stateDiagram-v2` 图，包含所有状态及其转换
- [ ] 错误状态有 `Then` 中的原文消息 — 非转述
- [ ] 成功状态描述用户看到/收到什么（重定向、消息等）
- [ ] 当 Scenarios 中有异步操作时描述了加载状态
- [ ] 默认状态描述用户首次访问时看到的内容

---

## Section 4 — 考量

**目的：** 记录从产物中推导的验证规则、可访问性、响应性和其他 UX 方面。

**建议的子 section：**

### 验证
从错误 Scenarios 和需求中推导的表单验证规则：
- `<字段>`: `<验证规则>`

### 可访问性
从上下文中识别或推断的可访问性方面（如：标签、焦点、屏幕阅读器）。

### 响应性
如果在 PRD 或 requirements 中提到，不同屏幕尺寸下的预期行为。

### 其他
以上未覆盖的屏幕特定考量。

**质量检查清单：**
- [ ] 验证规则从 Scenario Outline / Examples 和 requirements 推导
- [ ] 无可追溯到产物的假设
- [ ] 仅当在产物中有来源时才提及可访问性

---

## Section 5 — 视觉参考

**目的：** 为线框图、mockup 和原型保留空间，由团队手动填写。

**此 section 始终由 agent 留空**，仅包含以下 3 个占位符：

```markdown
## 视觉参考

### 线框图
_待手动填写。_

### Mockup
_待手动填写。_

### 交互原型
_待手动填写。_
```

**绝对规则：** `extract-views-agent` **绝不**在此 section 中填写内容。

## 一般格式规则

| 规则 | 值 |
|---|---|
| **输出文件** | `docs/features/<slug>/views/<规范名称>/tela.md` |
| **语言** | 中文。成熟的技术术语保持英文（token、cookie、upload 等） |
| **可追溯性** | 所有内容可追溯到 BDD Scenario、PRD 或 Stories |
| **无源代码** | 无任何代码片段、JSX、props 或库组件名称 |
| **标签** | 如场景和 PRD 中所示 — 绝不编造 |
| **错误消息** | `.feature` 中 `Then` 的原文副本 |
| **名称分隔符** | slugs 和文件夹名使用无重音的 kebab-case |
