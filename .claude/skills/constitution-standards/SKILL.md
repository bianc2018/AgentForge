# Constitution 标准

## 什么是 constitution.md

`constitution.md` 是一组不可协商的技术规则，约束项目或系统的所有实现。它充当团队与代码库之间的契约：任何违反 constitution 的实现都是无效的，必须在 merge 之前修正。

---

## 格式规范

> 如果项目中存在 `docs/sdd/constitution-format.md`，读取它并将其用作格式规范 — 它替代以下默认格式，可能已为此项目定制。

## 默认格式

### 必需结构

一个 constitution.md 由以下 section 组成，按此顺序：

1. Purpose（目的）
2. Must Do（必须做）
3. Ask Before Proceeding（继续前询问）
4. Never Do（绝不做）
5. Enforcement（执行）

---

## Section 和质量标准

### 1. Purpose

**目的：** 用 1-2 句话声明 constitution 是什么以及应如何使用。

**格式：** 连续文本（无 bullet，无表格）。

**质量检查清单：**
- [ ] 声明文档定义了不可协商的规则
- [ ] 提及范围（项目、系统、feature）
- [ ] 指示歧义必须明确解决 — 绝不假设
- [ ] 最多 3 句话

**质量足够的标志：** 任何开发人员无需阅读其余部分就能理解文档的角色。

---

### 2. Must Do

**目的：** 列出积极的技术义务 — 每个实现必须无例外地做什么。

**格式：** 编号列表。每项以"所有 [主体] 必须 [动词]"开头。

**质量检查清单：**
- [ ] 4 到 8 条规则
- [ ] 每条规则覆盖不同方面：分层、错误、验证、日志、可追溯性
- [ ] 每条规则可验证 — 可以判断"此实现遵循或不遵循此规则"
- [ ] 每条规则使用祈使语言（"必须"）
- [ ] 没有规则重复或重叠同一 section 或其他 section 中的另一条规则

**最低预期覆盖范围：**
- 分层分离（业务逻辑、传输、持久化）
- 使用标准化结构传播错误
- 在系统边界验证输入
- 关键操作日志记录
- 对需求或 BDD 的可追溯性

**质量足够的标志：** 代码审查者可以在审查 PR 时机械地检查每条规则。

---

### 3. Ask Before Proceeding

**目的：** 定义在做出明确决策前阻止开发的情况。

**格式：** 编号列表。每项以"如果 [条件]，[操作]。"开头。

**质量检查清单：**
- [ ] 3 到 6 条规则
- [ ] 每条规则有明确的触发条件（如果 X 则询问）
- [ ] 至少覆盖：需求不明确、具有多个选项的架构决策、契约/API 变更、与 constitution 本身的冲突
- [ ] 每个操作明确指示停止并寻求清晰度 — 不要带着假设继续
- [ ] 没有规则过于模糊而无法触发（如："如果有疑问"不行）

**质量足够的标志：** 开发人员可以无歧义地识别是否处于需要停止的情况。

---

### 4. Never Do

**目的：** 列出绝对禁止事项 — 任何实现绝不可以做什么。

**格式：** 编号列表。每项以"绝不 [动词]"开头。

**质量检查清单：**
- [ ] 4 到 8 条规则
- [ ] 每条规则是绝对禁止，无例外
- [ ] 至少覆盖：在错误层直接访问数据、领域外的业务逻辑、静默错误、与框架耦合、需求假设
- [ ] 没有规则与 Must Do 中的规则矛盾
- [ ] 每条规则描述具体的反模式，而非泛化

**质量足够的标志：** 任何开发人员在实现违反某条规则时能立即识别。

---

### 5. Enforcement

**目的：** 定义对违反 constitution 者的实际后果以及执行机制。

**格式：** 编号列表。每项描述后果或验证门。

**质量检查清单：**
- [ ] 2 到 4 条规则
- [ ] 至少一条规则要求实现计划描述对 constitution 的合规性
- [ ] 至少一条规则指定违规使实现无效，必须在 merge 前修正
- [ ] 至少一条规则在缺乏清晰度时阻止实现
- [ ] 后果是具体的（无效、阻止、要求修正）— 不模糊（建议、推荐）

**质量足够的标志：** 该 section 可作为可验证的预 merge 检查清单。

---

## 一般格式规则

- **语言：** 默认中文。技术术语（controller、repository、adapter、middleware）保持英文。
- **全局编号：** 规则在整个文档中顺序编号（1, 2, 3...）— 不按 section 重新开始。
- **无空 section：** 如果 section 不适用，明确记录原因。
- **不重复：** 如果一条规则已在一个 section 中说过，不要在其他 section 重复。
- **语气：** 客观直接。无需冗长的理由 — 规则应自解释。
- **分隔符：** 在 section 之间使用 `---` 以方便阅读。
