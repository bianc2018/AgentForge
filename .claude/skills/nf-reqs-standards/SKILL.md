# 非功能需求标准

## 格式规范

> 如果项目中存在 `docs/sdd/nf-reqs-format.md`，读取它并将其用作格式规范 — 它替代以下默认格式，可能已为此项目定制。

## 默认格式

### 什么是非功能需求

非功能需求（NFR）描述**系统在质量方面的行为方式**。

它不定义系统_做什么_，而是定义系统应以**何种质量**运行：
- 性能
- 可扩展性
- 可用性
- 安全性
- 可观测性
- 可用性

---

## 好的 NFR 的特征

NFR 应具备：

- **可测量**：用客观指标表达（如："< 2s"、"99.9% 可用率"）
- **可测试**：可以通过客观测试验证
- **具体**：与明确的上下文或操作绑定
- **可追溯**：从来源（PRD、Story、BDD、功能需求）推导而来

避免：
- 模糊术语："快速"、"安全"、"高可用"、"良好性能"

优先：
- 明确指标："2 秒内"、"95% 的请求"、"错误率 < 1%"

---

## NFR 类别

### 性能

系统操作的响应时间和吞吐量。

关键问题：
- 此操作的最大响应时间是多少？
- 在哪个百分位？（95%？99%？）
- 在正常负载还是峰值条件下？

---

### 可扩展性

支持用户或流量增长的能力。

关键问题：
- 系统应支持多少并发用户？
- 预期的流量增长是多少？
- 系统是否需要水平扩展？

---

### 可用性

正常运行时间和容错能力。

关键问题：
- 最低预期正常运行时间是多少？（99%？99.9%？99.99%？）
- 每月最大不可用时间是多少？
- 系统是否需要自动故障转移？

---

### 安全性

认证、授权、加密和数据保护。

关键问题：
- 哪些数据需要加密？
- 失败尝试的策略是什么（锁定、速率限制）？
- 哪些安全事件需要审计？

---

### 可观测性

日志、指标和追踪。

关键问题：
- 哪些事件必须记录到日志？
- 哪些指标必须被监控？
- 日志的最短保留时间是多少？

---

### 可用性

学习时间和可访问性。

关键问题：
- 是否有完成任务的最高时间要求？
- 是否有可访问性要求（WCAG）？
- 是否支持多语言？

---

## `nf-requirements.md` 文件的必需结构

```markdown
# 非功能需求 — <Feature 名称>

## 性能

**NFR-1**：在正常条件下，系统必须在 <条件> 下 <行为>，指标为 <指标>。

> 来源：<产物中的来源>

## 安全性

**NFR-2**：系统必须 <行为>。

> 来源：<产物中的来源>
```

---

## 每条 NFR 的格式

```
[可选条件，] 系统必须 [行为] [指标]。
```

良好示例：
- `在正常条件下，系统必须在 2 秒内响应认证请求，覆盖 95% 的案例。`
- `系统必须支持最多 5,000 个并发用户，无明显性能下降。`
- `系统必须具有每月最低 99.9% 的可用性。`
- `系统必须将所有认证尝试记录到日志，包括失败尝试。`
- `系统必须在连续 5 次登录失败后临时锁定账户。`

---

## 每条 NFR 的质量标准

### 可测量性

**检查清单：**
- [ ] NFR 包含客观指标（数字、百分比、时间、数量）
- [ ] 指标可以无歧义地验证
- [ ] 测量条件已指定（正常负载？峰值？百分位？）

---

### 可测试性

**检查清单：**
- [ ] 可以编写证明 NFR 已满足或违反的测试
- [ ] NFR 不使用模糊术语："快速"、"安全"、"高性能"、"高"
- [ ] 描述的行为可从外部观察

---

### 可追溯性

**检查清单：**
- [ ] `> 来源：` 字段标识 NFR 的来源（PRD、Story、BDD 或功能需求）
- [ ] NFR 与一致的类别关联

---

## NFR 集合的质量标准

| 标准 | 验证 |
|---|---|
| **按类别覆盖** | feature 相关的类别均已覆盖 |
| **可测量性** | 每条 NFR 都有客观指标 |
| **一致性** | 与 PRD 和功能需求词汇统一 |
| **非冗余** | 两条 NFR 不描述相同的质量约束 |
| **相关性** | 没有 NFR 过于通用或与 feature 无关 |

## 如何从产物中推导 NFR

### 从 PRD

| PRD 中的信号 | 产生的 NFR 类别 |
|---|---|
| "用户快速增长" | 可扩展性 |
| "系统必须始终可用" | 可用性 |
| "敏感数据"、"密码"、"token" | 安全性 |
| "实时"、"即时"、"无延迟" | 性能 |
| "追踪"、"审计"、"记录" | 可观测性 |
| "易于使用"、"可访问" | 可用性 |

### 从 User Stories

- 带有"快速"、"无等待"等词的验收标准 → 性能
- 带有"安全"、"受保护"、"已认证"的标准 → 安全性
- 带有"记录"、"通知"、"警报"的标准 → 可观测性

### 从 BDD 场景

- `Given N 个并发用户` → 可扩展性 / 性能
- `When 发起请求` + 时间预期 → 性能
- 带有重试或锁定的失败场景 → 安全性

### 从功能需求（EARS）

对每条功能需求，询问：
- 应该多快执行？→ 性能
- 同时有多少用户？→ 可扩展性
- 如果失败会发生什么？→ 可用性
- 哪些数据需要保护？→ 安全性
- 什么需要记录到日志？→ 可观测性

## 一般格式规则

- **输出文件：** `docs/features/<slug>/nf-requirements.md`
- **NFR 语言：** 中文
- **编号：** 全局顺序 — NFR-1, NFR-2, NFR-3...（不按类别重新开始）
- **分组：** 按类别使用 `##` 标题
- **分隔：** NFR 之间空一行
