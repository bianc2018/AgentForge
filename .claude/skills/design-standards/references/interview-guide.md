# 访谈指南 — 技术设计创建

## 访谈原则

1. **一次一个问题。** 绝不在同一条消息中问两个问题。

2. **推导先于提问。** 现有产物是首要来源：
   - `requirements.md` → 必需行为（什么）
   - `nf-requirements.md` → 技术约束（如何行为）
   - `scenarios.feature` → 执行流程和 API 契约（流程骨架和错误响应）
   - `stories.md` → 验收标准和输入验证
   - `prd.md` → 外部依赖、风险和不在范围内
   - `doc/constitution.md` → 不可协商的全局架构约束

3. **开放式问题，非确认性问题。** 优先使用"你如何设想与 SMS 服务的集成？"而非"你要用 Twilio 吗？"。

4. **使用累积的上下文。** 如果 NFR 已定义"5 次尝试，锁定 30 分钟"，不要问关于速率限制的问题。

5. **模糊时深入挖掘。** 如果回答是"标准做法"或"常规做法"，要求具体选择。

---

## 何时接受 vs. 深入挖掘

**接受**当：
- Section 检查清单中的所有项目已被产物或回答覆盖
- 信息具体且可验证
- 生成的草稿没有明显空白

**深入挖掘**当：
- 回答模糊（"能用的东西"、"标准做法"、"看情况"）
- 存在未处理的真正权衡
- 决策与已提出的需求矛盾
- Section 检查清单中至少缺失一个必需项目

## 各 section 的问题库

### 1. 技术概述

当草稿对所选择的技术方案不清楚时使用。

- "这里的核心技术方案是什么？（如：数字 OTP、魔法链接、安全问题）"
- "`doc/constitution.md` 中是否有直接影响此 feature 的约束？"
- "此 feature 是否适合项目中已建立的某种模式，还是新东西？"

### 2. 组件架构

当组件的职责不明确或需要新的外部服务时使用。

- "此组件需要是新的，还是项目中已有可以扩展的？"
- "此组件的单一职责是什么？用一句话。"
- "是否涉及外部服务（邮件、短信、推送）？是否已选择 provider 还是待定？"
- "该外部服务应隔离在 adapter 中，还是项目中已有抽象？"

### 3. 数据模型

当字段、类型或关系不能从产物中推导时使用。

- "此实体需要软删除还是可以在使用后物理删除？"
- "这里哪些审计字段是必需的？（created_at、updated_at、created_by 等）"
- "OTP 验证码应存储到数据库还是 Redis？"
- "此实体是新的还是项目中现有实体的扩展？"

### 4. API/契约

当 API 约定未在产物中声明时使用。

- "密码恢复的 endpoints 是公开的还是需要某种预先认证？"
- "项目的标准错误格式是什么？（如：`{ error: string, code: string }`）"
- "API 是否版本化？使用什么前缀？（如：`/api/v1/`）"

### 5. 执行流程

仅当存在 BDD 场景未覆盖的内部行为时使用。

- "如果邮件或 SMS 服务在发送时不可用，应该发生什么？（静默失败还是用户错误？）"
- "此步骤可以是异步的（队列）还是必须是同步的以满足 SLA？"

### 6. 技术决策

对每个未被产物解决的真正权衡使用。

- "数字 OTP 还是魔法链接？支持 SMS 的事实是否影响这个选择？"
- "速率限制计数器是放在数据库中还是 Redis 中？"

## 常见错误要避免

| 错误 | 正确 |
|------|---------|
| 问 PRD 已有的内容 | 从 PRD 推导并在草稿中展示 |
| 问 NFR 已定义的内容 | 直接在草稿中应用 |
| 问 `doc/constitution.md` 已定义的内容 | 应用并在草稿中提及 |
| 编写执行流程时忽略 BDD 场景 | 使用每个 Scenario 作为流程骨架 |
| 在设计中使用代码（函数名、SQL 查询） | 描述组件和职责 |
| 混淆"什么"（需求）与"如何"（设计） | 设计只回答"如何" |
| 创建组件而不追溯到需求 | 每个组件必须处理至少一个 REQ |
| 在技术决策中列出显而易见的决策 | 仅包含真正的权衡 |
