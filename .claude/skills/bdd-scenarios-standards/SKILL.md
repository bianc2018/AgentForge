# BDD 场景标准

## 格式规范

> 如果项目中存在 `docs/sdd/scenarios-format.md`，读取它并将其用作格式规范 — 它替代以下默认格式，可能已为此项目定制。

## 默认格式

### `.feature` 文件的必需结构

每个文件遵循以下格式：

```gherkin
Feature: <Feature 名称>
  <可选的一行描述，说明 feature 的目的>

  Scenario: <场景标题>
    Given <前置条件或系统状态>
    When <用户执行的操作或事件>
    Then <预期的可观察结果>
    And <附加结果，如有必要>

  Scenario: <另一个场景>
    ...
```

---

## 场景各部分的质量标准

### `Feature`

**目的：** 标识正在描述的功能。

**质量检查清单：**
- [ ] Feature 名称与 PRD 和 User Stories 一致
- [ ] 描述（可选）面向业务，而非面向实现

---

### `Scenario`

**目的：** 描述系统的一个可观察、可验证的行为。

**质量检查清单：**
- [ ] 标题具有描述性和特定性 — 无需阅读步骤即可识别正在测试的内容
- [ ] 每个场景描述**一个单一行为**（不在同一场景中混合正常路径和错误）
- [ ] 集合包括：1 个成功场景（正常路径）、2–3 个错误场景、至少 1 个边缘案例
- [ ] 错误场景标题使用模式：`<操作> 在 <失败条件> 下`（如："使用错误密码登录"）

---

### `Given`

**目的：** 描述操作前系统的初始状态。

**质量检查清单：**
- [ ] 描述系统状态或前置条件，而非用户操作
- [ ] 使用领域语言，而非技术实现术语
- [ ] 不提及数据库、JWT、bcrypt、表或库
- [ ] 如果前置条件是显然的（如："系统可用"），可以省略以保持场景简洁

---

### `When`

**目的：** 描述触发场景中行为的操作。

**质量检查清单：**
- [ ] 描述用户的单一操作或外部事件
- [ ] 使用现在时态的动作动词（"配送员发送"、"系统收到"）
- [ ] 不提及实现（不使用"调用 endpoint"、"发起 POST"、"执行查询"）

---

### `Then`

**目的：** 描述操作后预期的可观察结果。

**质量检查清单：**
- [ ] 从用户或外部观察者的角度描述结果
- [ ] 可验证 — 观察系统时可以判断"通过"或"失败"
- [ ] 不提及外部不可观察的系统内部状态
- [ ] `And` 仅用于同一场景的附加结果，不用于新操作

## 场景集的质量标准

一个健康的 feature 场景集应满足：

| 标准 | 验证 |
|---|---|
| **覆盖率** | 覆盖正常路径、主要错误和至少一个边缘案例 |
| **清晰性** | QA 仅通过阅读场景就能执行测试 |
| **独立性** | 每个场景独立运行，不依赖执行顺序 |
| **领域语言** | 团队中的任何人都能理解场景，无需了解代码 |
| **可追溯性** | 每个场景在 PRD（主流程、替代流程、成功标准）或 User Stories 中有明确来源 |

## 场景类型及何时创建

### 成功场景（happy path）
覆盖一切按预期进行时的主流程。
**来源：** PRD 的**主流程** section 或正常路径故事的验收标准。

### 输入错误场景
覆盖用户提供无效或不完整数据时的行为。
**来源：** PRD **目标** section 中带有"拒绝"、"验证"动词的项目，或验证故事的验收标准。

### 无效状态场景
覆盖系统处于意外状态或前置条件未满足时的行为。
**来源：** PRD 的**替代流程** section。

### 边缘案例场景
覆盖既不是正常路径也不是常见错误的边界情况。
**来源：** PRD 的**风险** section 或安全故事的验收标准。

## 一般格式规则

- **输出文件：** `docs/features/<slug>/scenarios.feature`
- **缩进：** `Scenario` 内的步骤使用 2 个空格
- **步骤语言：** 中文。成熟的技术术语保持英文（token、JWT、cookie、OAuth）
- **分隔：** 场景之间空一行
- **步骤中无内联注释** — 步骤应当自解释
- **无 `Background`** — 除非用户明确要求 — 为清晰起见，优先在每个场景中使用显式的 `Given`
