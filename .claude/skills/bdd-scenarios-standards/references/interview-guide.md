# 访谈指南 — BDD 场景创建

## 访谈原则

1. **一次一个问题。** 绝不在同一条消息中问两个问题。

2. **PRD 和 Stories 是首要来源。** 在提问之前尽可能多地进行推导。问题是针对产物无法回答的空白。

3. **开放式问题，非确认性问题。** 优先使用"当配送员已有活跃会话时会发生什么？"而非"在这种情况下我们应该重定向到仪表盘吗？"。

4. **关注可观察的行为。** 如果候选场景描述系统内部状态（数据库、JWT、token），询问这对用户如何体现。

5. **不要创建超出范围的场景。** 如果用户为 PRD 中"不在范围内" section 的内容要求一个场景，引用该段落礼貌拒绝。

---

## 何时接受 vs. 深入挖掘

**接受**当：
- 场景检查清单中的所有项目已覆盖
- 标题清楚地标识了行为而无需阅读步骤
- `Given`、`When` 和 `Then` 使用领域语言，无技术细节
- 结果（`Then`）可观察且可验证

**深入挖掘**当：
- `Then` 描述内部状态（"系统保存到数据库"、"token 被失效"）
- `Given` 模糊（"假设系统正在运行"）
- `When` 混合了两个不同的操作
- 标题没有将此场景与其他场景区分开（"登录场景"）
- 单个场景同时涵盖正常路径和错误

## 场景各部分的问题库

### `Given` — 前置条件

当初状态模糊或缺失时使用。

- "在此操作发生之前，配送员或系统的情况是什么？"
- "这个故事是否预设了 PRD 中的某个前置条件（如：账户已创建、会话活跃）？"
- "这个场景适用于任何配送员还是仅适用于具有特定特征的配送员？"

### `When` — 操作

当操作以技术术语描述或不明确时使用。

- "配送员如何执行此操作？他在界面上做什么？"
- "此操作是由用户发起还是系统事件？"
- "在此上下文中，什么具体特征使此操作无效/有效？"

### `Then` — 结果

当结果不可观察或通用时使用。

- "当此操作发生时，配送员看到或收到什么作为响应？"
- "QA 仅通过观察系统如何知道此行为发生了？"
- "除了主要消息，是否有任何可见的副作用（重定向、通知、屏幕状态变化）？"

### 场景覆盖率

当集合看起来不完整时使用。

- "PRD 提到了 [替代流程 X]。该流程已经被某个场景覆盖还是需要一个新的？"
- "有没有 QA 团队可能会尝试测试但尚未映射的边缘案例？"
- "故事 [N] 的验收标准是否全部被至少一个场景覆盖？"

## 警告信号 — 需要深入挖掘的草稿

| 观察到的问题 | 如何深入挖掘 |
|---|---|
| `Then` 提及数据库或 JWT | "配送员如何从外部感知这个结果？" |
| `Given` 描述一个操作（"假设配送员点击了..."） | "这一步是一个操作 — 可以移到 `When` 或成为一个单独的场景吗？" |
| 通用标题（"成功场景"） | "这个场景与其他场景的区别是什么？标题应标识具体行为。" |
| `When` 有两个动作动词（"发送并等待"） | "这是两个独立的操作吗？是否适合拆分为两个场景？" |
| 当上下文相关时，场景没有 `Given` | "此行为发生需要满足什么前置条件吗？" |
| 集合中没有边缘案例场景 | "有没有尚未覆盖的边界情况（会话过期、验证码已使用、字段为空）？" |

## 访谈节奏

- 始终宣布正在构建哪个场景或组：`[场景 3/<N>：<标题>]`
- 在提问之前展示生成的 Gherkin 草稿
- 纳入回答后，展示更新的草稿
- 当场景准备好时：`✅ 场景 <N> 已完成。`
- 无阻力地接受跳过或删除场景的请求；调整隐式编号

---

## 如何从 PRD 和 User Stories 推导场景

### 从 PRD 的主流程

- 整个主流程 → 1 个成功场景（正常路径）
- 流程中描述的每个用户输入 → 检查是否需要独立的场景

### 从 PRD 的替代流程

- 每个替代流程 → 1 个无效状态或错误场景

### 从 PRD 的成功标准

- 每个可测量的标准 → 检查是否有验证它的场景

### 从 User Stories

- 每个验收标准 → 至少 1 个覆盖它的场景
- 验证标准（"系统必须拒绝..."） → 专用的错误场景
- 安全标准 → 边缘案例或限制场景

### 从 PRD 的不在范围内 section

- 用作**负面过滤**：任何场景都不应涵盖此处列出的项目
