# 访谈指南 — 用户故事创建

## 访谈原则

1. **一次一个问题。** 绝不在同一条消息中问两个问题。用户需要回答多件事时会失去思路。

2. **PRD 是首要来源。** 在提问任何事之前，尽可能多地从 PRD 推导。问题是针对 PRD 无法回答的空白。

3. **开放式问题，非确认性问题。** 优先使用"这里对配送员的具体好处是什么？"而非"好处是能够继续使用系统吗？"。

4. **用 PRD 进行上下文关联。** 如果 PRD 将目标用户定义为"通过 register-user 创建了账户的配送员"，不要问"谁使用这个 feature？"。

5. **模糊时深入挖掘。** 如果用户说"验证数据"，问"具体是哪些数据，在这种情况下什么构成无效数据？"。

6. **不要创建超出范围的故事。** 如果用户为 PRD 中"不在范围内" section 的内容要求一个故事，引用 PRD 的段落礼貌拒绝。

---

## 何时接受 vs. 深入挖掘

**接受**当：
- 故事检查清单中的所有项目已覆盖
- 验收标准具体且可验证
- Persona 和收益与 PRD 一致

**深入挖掘**当：
- "以便"琐碎或通用（"以便使用系统"、"以便工作"）
- 验收标准模糊（"系统必须正常工作"）
- Persona 通用（"用户"、"某人"）
- 标准提及技术实现细节
- 故事同时涵盖两个不同的流程（应拆分）

---

## 故事各部分的问题库

### 标题 — "作为"

当 persona 不清楚或通用时使用。

- "PRD 将目标用户定义为 [X]。这个故事是针对同一用户，还是涉及不同的 persona？"
- "配送员需要满足什么前置条件才能到达流程中的这一点？"

### 标题 — "我想要"

当操作以技术术语描述或模糊时使用。

- "配送员将如何体验这个能力？他会做什么或看到什么？"
- "如果你要用一句话向非技术人员描述配送员通过这个能做什么，会怎么说？"

### 标题 — "以便"

当收益通用或 PRD 中未明确时使用。

- "当这个故事实现后，对配送员的具体结果是什么？对他来说有什么改变？"
- "为什么这在他的工作背景下对配送员很重要？"

### 验收标准 — 正常路径

当正常路径标准缺失或模糊时使用。

- "当这个故事中一切按预期进行时，配送员看到或收到什么？"
- "系统如何确认配送员的操作成功了？"

### 验收标准 — 错误/验证场景

当没有失败或验证标准时使用。

- "当 [与此故事相关的无效数据或错误条件] 时会发生什么？"
- "PRD 提到了 [拒绝/验证目标]。这需要作为这个故事的标准出现吗？"

### 验收标准 — 具体性

当标准模糊或不可验证时使用。

- "'[模糊标准]' 在这种情况下是什么意思？你能给出一个具体的'通过'和'失败'的例子吗？"
- "开发人员能否按现状为这个标准编写测试？他具体会验证什么？"

### 故事拆分

当故事似乎涵盖两个不同的流程时使用。

- "这个故事涵盖了 [流程 A] 和 [流程 B]。将它们分开以使每个更聚焦是否合理？"
- "[流程 B] 需要在 [流程 A] 之前实现，还是它们是独立的？"

## 警告信号 — 需要深入挖掘的回答

| 收到的回答 | 问题 | 如何深入挖掘 |
|---|---|---|
| "以便我可以使用系统" | 琐碎的收益 | "这在日常工作中为配送员解决了什么具体问题？" |
| "系统必须正常工作" | 不可验证的标准 | "在这里'正常'是什么意思？什么会是成功和失败的例子？" |
| "验证数据"（未指定哪些） | 模糊的标准 | "具体是哪些数据？在这种情况下什么使它们无效？" |
| "用户"或"某人"（在 persona 中） | 通用 persona | "这是 PRD 中定义的同一个配送员，还是涉及其他类型的用户？" |
| 标准提及 JWT、bcrypt、表 | 技术细节 | "配送员如何从外部感知到这一点，而不知道它是如何实现的？" |
| 一个有 7+ 个标准的故事 | 故事太大 | "这些标准是否涵盖多个流程？可能适合拆分为两个故事。" |

## 访谈节奏

- 始终宣布正在构建哪个故事：`[故事 2/5：数据验证]`
- 在提问之前展示生成的草稿
- 纳入回答后，展示更新的草稿并在进一步提问或推进之前大声评估
- 当故事准备好时，确认：`✅ 故事 <N> 已完成。前进到 [下一个故事]。`
- 如果用户想跳过或从索引中删除故事，接受并调整后续编号

## 如何从 PRD 推导故事

### 从目标 section

- 带有动词 **"允许"** 的目标 → 正常路径故事（用户能做到什么）
- 带有动词 **"拒绝"** 的目标 → 验证故事（系统阻止什么）
- 带有动词 **"验证"** 或 **"检测"** 的目标 → 验证故事
- 与同一流程相关的过于细粒度的目标 → 可以合并为一个故事

### 从主流程 section

- 流程中每个**重要的用户输入**（非每个系统技术步骤）倾向于成为一个故事
- 纯粹的系统内部步骤（如："系统验证 token"、"系统签发 JWT"）出现在验收标准中，而非作为独立故事

### 从替代流程 section

- 每个对**用户有可见影响**的替代流程 → 失败案例故事
- 没有用户操作的纯技术替代流程 → 不成为故事

### 从风险 section

- **缓解措施对用户可见**的风险（如：过度尝试锁定） → 安全故事
- 纯技术风险（如："使用 256 位 secret 签名的 JWT"） → 不成为故事

### 从不在范围内 section

- 仅用作**负面过滤**：如果候选故事涵盖此处列出的内容，放弃它并通知用户
