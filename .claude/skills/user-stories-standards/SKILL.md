# 用户故事标准

## 格式规范

> 如果项目中存在 `docs/sdd/user-stories-format.md`，读取它并将其用作格式规范 — 它替代以下默认格式，可能已为此项目定制。

## 默认格式

### 每条故事的必需结构

每条故事遵循以下格式，无例外：

```markdown
## 故事 <N> – <标题>

作为 <persona>
我想要 <操作或能力>
以便 <收益或结果>

_验收标准_：

- <标准 1>
- <标准 2>
- <标准 N>
```

---

## 故事各部分的质量标准

### 标题（`作为 / 我想要 / 以便`）

**目的：** 从使用者而非构建者的角度表达用户需求。

**质量检查清单：**
- [ ] **作为** — 是特定的 persona，非通用的（"已创建账户的配送员"，而非"用户"或"某人"）
- [ ] **作为** — persona 与 feature PRD 的目标用户 section 一致
- [ ] **我想要** — 表达用户的能力或操作，而非技术实现细节
- [ ] **我想要** — 使用业务语言（"保持登录状态"），而非技术语言（"收到 refresh token"）
- [ ] **以便** — 表达对用户的实际收益或具体结果
- [ ] **以便** — 不是显而易见的琐碎内容（避免"以便我可以使用系统"）
- [ ] "作为 / 我想要 / 以便" 三者连贯阅读时形成连贯的语句

**质量足够的标志：** 团队中的任何人都能理解谁需要什么以及为什么，无需阅读标准。

---

### 验收标准

**目的：** 以可验证的方式定义系统必须做什么才能认为故事已完成。

**格式：** Bullet 列表。每项以"系统必须..."或"系统不得..."开头。

**质量检查清单：**
- [ ] 每条故事 2 到 6 个标准
- [ ] 每个标准可验证 — 可以在测试中判断"通过"或"失败"
- [ ] 覆盖正常路径（一切顺利时会发生什么）
- [ ] 覆盖至少一个与此故事相关的验证或错误场景
- [ ] 没有标准字面重复故事的"我想要"
- [ ] 没有标准提及实现细节（函数名、数据库结构、库）
- [ ] 标准与 PRD 一致 — 不与目标、流程或不在范围内矛盾

**质量足够的标志：** 开发人员可以根据每个标准编写自动化测试，无需询问任何问题。

## 故事集的质量标准（改编版 INVEST）

根据 INVEST 的每个字母评估完整集合：

| 字母 | 原则 | 验证 |
|---|---|---|
| **I** | 独立 | 每条故事可以独立实现，不依赖其他故事的顺序（除 PRD 中的显式前置条件外） |
| **N** | 可协商 | "如何"和"为什么"已与用户验证 — 不是由 agent 编造的 |
| **V** | 有价值 | 每条故事为最终用户交付可感知的价值，不仅仅是技术基础设施 |
| **E** | 可估算 | 开发人员可以根据标准估算实现工作量 |
| **S** | 小 | 每条故事覆盖单个流程或职责；不同流程是独立的故事 |
| **T** | 可测试 | 所有验收标准可通过测试（手动或自动化）验证 |

## 故事类型及何时使用

### 正常路径故事（happy path）
覆盖一切按预期进行时的主用户流程。
PRD 来源：**主流程** section 或**目标** section 中带有"允许"动词的项目。

### 验证故事
覆盖收到无效或意外数据时系统的行为。
PRD 来源：**目标** section 中带有"验证"、"拒绝"或"检测"动词的项目。

### 失败案例故事
覆盖用户可见的错误或异常情况下系统的行为。
PRD 来源：**替代流程** section。

### 安全/限制故事
覆盖访问控制、数据保护或滥用尝试行为。
PRD 来源：**风险**或**成功标准** section，当它们提及对用户有可见影响的安全时。

## 一般格式规则

- **Section 标题：** `## 故事 <N> – <标题>`（使用破折号 `–`，非连字符 `-`）
- **编号：** 从 1 开始顺序编号，不跳号
- **语气：** 面向用户，非面向实现。PM 不需要懂代码就能理解所有故事。
- **语言：** 中文。成熟的技术术语保持英文（JWT、cookie、OAuth、bcrypt）。
- **文件标题：** `# 用户故事 — <Feature 名称>`（使用破折号）
- **分隔：** 故事之间空一行
- **标准：** 绝不让故事没有验收标准
