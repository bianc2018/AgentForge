# 访谈指南 — 功能需求创建（EARS）

## 访谈原则

1. **一次一个问题。** 绝不在同一条消息中问两个问题。

2. **三个产物是首要来源。** 在提问之前尽可能多地从 PRD、Stories 和 BDD 场景推导。问题是针对产物无法回答的空白。

3. **开放式问题，非确认性问题。** 优先使用"当注册期间会话过期时会发生什么？"而非"在这种情况下系统应该重定向到登录吗？"。

4. **关注系统义务，而非用户行为。** 需求描述了系统**必须做什么**，而非用户做什么。

5. **不要为超出范围的项目创建需求。** 如果用户为 PRD 中"不在范围内" section 的内容要求一个需求，引用该段落礼貌拒绝。

---

## 何时接受 vs. 深入挖掘草稿

**接受**当：
- 需求对该行为类型使用了正确的 EARS 模式
- 动词 `shall` 存在
- 行为可通过测试验证
- 没有提及技术、库或框架
- 词汇与 PRD 一致

**深入挖掘**当：
- 行为模糊（"系统必须处理错误"）
- 需求提及技术（"the system shall store the JWT in Redis"）
- EARS 模式错误（如：对失败条件使用 `When`，应该使用 `If`）
- 需求描述了用户做什么，而非系统做什么
- 行为不能通过测试验证
- 两个不同的行为在同一个需求中

## 各类型空白的问题库

### EARS 模式不正确

当草稿对上下文使用了错误的模式时使用。

- "这个行为是否始终发生（普适）、当某事发生时（事件）、当状态持续时（状态）、或当某事失败时（不期望）？"
- "这是错误行为还是正常流程行为？这决定了我们使用 `If` 还是 `When`。"

### 行为模糊

当 `shall <behavior>` 过于通用时使用。

- "当这种情况发生时，系统具体必须做什么？可观察的响应是什么？"
- "QA 仅通过测试系统如何知道这个需求已被满足？"
- "是否有特定的消息、重定向、可见状态代表此行为？"

### 条件/事件不够精确

当 `When <event>` 或 `If <condition>` 模糊时使用。

- "什么具体特征定义了此事件？什么将这个场景与其他类似场景区分开来？"
- "这个条件有变体吗？例如，空字段与无效格式不同吗？"
- "触发此系统行为的确切触发器是什么？"

### 技术被提及

当需求提及实现时使用。

- "你如何在不提及它是如何实现的情况下描述此行为？"
- "从业务角度来看，系统必须做什么 — 不引用数据库、token 或库？"

### 集合覆盖率

当需求集看起来不完整时使用。

- "BDD 场景 '[标题]' 还没有对应的需求。它隐含了系统的什么义务？"
- "故事 [N] 的验收标准 '[标准]' 被任何需求覆盖了吗？如果没有，它定义了哪些必需行为？"
- "PRD 中是否有尚未产生 `If` 需求的失败条件？"

---

## 何时接受"N/A"回答

如果用户说某些东西不适用（如："我们不需要为此建立需求"）：
- 内部记录并前进到下一个项目
- 不要坚持
- 如果是明显的空白（如：没有任何错误场景有需求），提及一次并接受决定

---

## 访谈节奏

- 始终宣布正在构建哪个需求：`[需求 X/<N>：<标题>]`
- 在提问之前展示生成的 EARS 草稿
- 纳入回答后，展示更新的草稿
- 当需求准备好时：`✅ 需求 <N> 已完成。`
- 无阻力地接受跳过或删除需求的请求；调整隐式编号

---

## 如何从 BDD 场景推导需求（逐步流程）

对于每个场景 `Scenario: <标题>`：

1. 读取 `When` — 这是**事件**或**条件**
2. 读取 `Then` — 这是**系统的预期行为**
3. 选择 EARS 模式：
   - 场景的 `When` 是正常操作 → 模式 `When <event>, the system shall <behavior>`
   - 场景的 `When` 是失败条件 → 模式 `If <condition>, the system shall <behavior>`
4. 如果 `Then` 有 `And`，评估附加行为是否值得独立的需求
5. 用领域词汇以英文编写需求

### 推导示例

BDD 场景：
```gherkin
Scenario: Successful registration
  Given a new delivery driver on the registration form
  When valid registration data is submitted
  Then the driver account is created
  And the driver receives a confirmation email
```

推导的需求：
```
REQ-1: When valid registration data is submitted, the system shall create a delivery driver account.
REQ-2: When a delivery driver account is successfully created, the system shall send a confirmation email to the registered address.
```

## 警告信号 — 需要深入挖掘的草稿

| 观察到的问题 | 如何深入挖掘 |
|---|---|
| `shall` 缺失（如："the system should"） | "我们使用 `shall` 表示义务。此行为是强制性的吗？" |
| 需求中有技术（"shall save to MySQL"） | "如何在不提及持久化技术的情况下描述这一点？" |
| 行为模糊（"shall handle errors"） | "当错误发生时，系统必须具有什么具体行为？" |
| `When` 用于失败条件 | "这是一个错误场景 — 应使用 `If` 而非 `When`。" |
| 一个需求中有两个行为 | "这是两个不同的行为。是否适合拆分为两个需求？" |
| 需求描述用户操作 | "需求必须描述系统做什么，而非用户做什么。" |
